# AWS 아키텍처 설계 — 웹 이벤트 파이프라인

## 구성도 레이어 요약

```
[생성]  Web/Mobile  →  EC2 (generator)        ← ECR (이미지)
           ↓
[수집]  API Gateway  →  Kinesis Data Streams  →  Kinesis Firehose
                                                       ↓
[저장]  S3 (Raw)  ←──────────────────────────────────-┘
            ↓ Lambda ETL
        RDS Aurora (PostgreSQL)    S3 (Processed / 차트)
           ↓
[분석]  AWS Glue  →  Athena   ECS Fargate (analytics.py)
                                       ↓
[시각화]  QuickSight    S3+CloudFront    ECS Fargate (Streamlit)

[운영]  CloudWatch  │  IAM  │  VPC  │  Secrets Manager
```

## AWS 서비스 구성 및 선택 이유

### 생성 레이어

| 서비스 | 역할 | 선택 이유 |
|---|---|---|
| EC2 | generator.py 컨테이너 실행 | 장기 실행 프로세스를 안정적으로 유지, 인스턴스 크기 자유 조정 |
| ECR | 컨테이너 이미지 저장소 | ECS/EC2에서 바로 pull 가능, IAM으로 접근 제어 |

### 수집 레이어

| 서비스 | 역할 | 선택 이유 |
|---|---|---|
| API Gateway | REST 엔드포인트 | 클라이언트가 HTTP로 이벤트를 푸시할 때 진입점. 인증·스로틀링 내장 |
| Kinesis Data Streams | 실시간 스트림 버퍼 | 초당 수만 건 이벤트를 유실 없이 받아 downstream에 전달 |
| Kinesis Firehose | S3 자동 적재 | 버퍼링 + 배치 압축(parquet/gzip)으로 S3에 자동 전송, 코드 없음 |

### 저장 레이어

| 서비스 | 역할 | 선택 이유 |
|---|---|---|
| S3 (Raw zone) | 원본 이벤트 파일 보관 | 무한 확장, 저렴한 비용, Athena와 직접 연동 |
| RDS Aurora (PostgreSQL) | 구조화된 필드 단위 저장 | 기존 과제와 동일한 PostgreSQL 방언 유지, Multi-AZ 자동 장애복구 |
| Lambda | ETL 변환 | S3 트리거로 서버리스 실행, 짧은 파싱·검증 작업에 최적 |
| S3 (Processed) | 집계 결과 및 차트 PNG 보관 | CloudFront 오리진으로 사용, 정적 콘텐츠 서빙 비용 최소화 |

### 분석 레이어

| 서비스 | 역할 | 선택 이유 |
|---|---|---|
| AWS Glue | ETL 스케줄링 + 데이터 카탈로그 | Athena가 S3 데이터 구조를 인식하려면 카탈로그 필요. 배치 집계 자동화 |
| Athena | S3 위 SQL 집계 | 서버 없이 SQL만으로 parquet 분석, 사용한 데이터량만 과금 |
| ECS Fargate | analytics.py 실행 | 서버 관리 없이 컨테이너 실행, 분석 완료 후 자동 종료로 비용 절감 |

### 시각화 레이어

| 서비스 | 역할 | 선택 이유 |
|---|---|---|
| QuickSight | BI 대시보드 | Athena·RDS 직접 연결, 드래그앤드롭 차트 구성, 추가 서버 불필요 |
| S3 + CloudFront | 차트 PNG 정적 서빙 | analytics.py가 생성한 PNG를 CDN으로 배포, 레이턴시 최소화 |
| ECS Fargate | Streamlit 앱 | Python 기반 인터랙티브 대시보드, 기존 코드 재사용 가능 |

### 운영 레이어

| 서비스 | 역할 | 선택 이유 |
|---|---|---|
| CloudWatch | 메트릭·로그·알람 | EC2/ECS/RDS/Lambda 로그를 한 곳에서 수집, 에러율 임계 알람 |
| IAM | 권한 관리 | 서비스 간 최소 권한 원칙 적용, 자격증명 코드 미포함 |
| VPC | 네트워크 격리 | RDS·Fargate를 프라이빗 서브넷에 배치, 외부 직접 접근 차단 |
| Secrets Manager | DB 자격증명 보관 | 코드에 패스워드 하드코딩 없이 런타임에 안전하게 주입 |

## AWS 서비스 역할 차이

### Kinesis Data Streams vs Kinesis Firehose

Data Streams는 이벤트를 실시간으로 버퍼링하면서 여러 consumer가 각각 독립적으로 읽을 수 있는 구조입니다. 대신 consumer 설계와 shard 관리까지 직접 책임져야 해서 운영 복잡도가 올라갑니다.

Firehose는 이보다 단순한 구조로, 스트림 데이터를 일정 조건(시간/크기)에 따라 자동으로 S3나 Redshift로 적재합니다. 별도 consumer 로직이 필요 없지만, 실시간 처리 유연성은 제한됩니다.

이번 설계에서는 **“실시간 처리 제어”보다는 “안정적인 S3 적재와 운영 단순성”을 우선**해서 Firehose를 함께 사용했습니다.

### S3 vs RDS Aurora

**S3**는 파일 단위 오브젝트 스토리지로, 구조화되지 않은 원본 데이터를 장기 저장하는 데 적합하고, 비용 효율이 매우 높으며 용량 제한이 사실상 없고 저렴하다는 장점이 있습니다. 대신 SQL로 필드를 조회하려면 Athena 같은 도구가 별도로 필요합니다.

**RDS Aurora**는 정형 데이터를 대상으로 빠른 조회와 트랜잭션 및 집계 처리가 가능한 구조의 관계형 DB로 인덱스·트랜잭션·조인이 전부 가능하다는 장점이 있습니다. 단, 용량을 미리 프로비저닝해야 하고 S3보다 비쌉니다.

여기서는 “원본 재처리 가능성”과 “분석용 빠른 조회”를 동시에 만족시키기 위해 S3 + RDS를 분리했습니다. 단일 저장소로 통합하는 방식도 가능하지만, 운영 안정성 측면에서는 이중화가 더 적절하다고 판단했습니다.

### Lambda vs ECS Fargate

**Lambda**는 이벤트 기반 단기 실행에 최적화되어 있으며, 실행 시간 제한(최대 15분)과 패키지 크기 제약이 존재하여 이벤트 하나가 들어오면 함수 하나가 실행되고 끝나며 상태를 유지할 수 없습니다. S3에 파일이 하나 올라올 때마다 파싱하는 ETL 같은 짧은 작업에 적합합니다.

**ECS Fargate**는 컨테이너 단위로 실행되기 때문에 실행 시간 제한이 없고, 기존 Docker 기반 분석 로직을 그대로 재사용할 수 있습니다. analytics.py처럼 DB에 연결해서 쿼리 4개를 돌리고 차트를 그리는 작업은 수 분이 걸릴 수 있는데, 이런 배치 작업이나 항상 켜져 있어야 하는 Streamlit 앱에 적합합니다. 대신 Lambda보다 초기 구조가 무겁고, 항상 “어떤 작업을 컨테이너로 가져갈지”에 대한 설계가 필요합니다.

### Athena vs RDS (쿼리 관점)

**Athena**는 S3에 있는 parquet 파일을 SQL로 쿼리합니다. 데이터를 직접 스캔하는 방식이라 인프라 관리가 필요 없고 사용한 데이터량 기준으로 과금됩니다. 매 쿼리마다 스캔 비용이 발생하므로(TB당 약 $5)으로만 과금됩니다. 매번 S3를 스캔하므로 빠른 응답이 필요하거나 대용량 데이터 서비스에는 부적합할 수 있습니다.

**RDS**는 이미 인덱스가 걸려 있는 테이블에서 일반적인 OLTP 쿼리가 가능해 응답이 밀리초 단위로 빠르지만, 항상 프로비저닝된 리소스를 유지해야 한다는 고정 비용 구조가 있습니다.

그래서 **Athena는 대규모 배치 분석, RDS는 실시간성 있는 조회**에 각각 역할을 분리했습니다.

## 설계에서 고민한 부분

### 1. 수집 경로

처음에는 API Gateway에서 바로 RDS로 INSERT하는 단순 구조도 고려했습니다.

하지만 직접 쓰는 방식은 구현이 단순한 반면, 트래픽이 갑자기 몰릴 때 RDS 커넥션 풀이 고갈되거나 DB가 병목이 됩니다. 

이때, Kinesis를 중간에 버퍼로 두면 downstream이 자기 속도로 소화할 수 있습니다. 또 Firehose를 함께 사용해 S3에 원본 데이터를 동시에 적재하도록 구성했는데, 이는 나중에 RDS 스키마가 바뀌더라도 원본 데이터를 다시 재처리할 수 있는 안전망이 될 수 있습니다.

### 2. 저장소 이중화

S3에만 저장하고 Athena로 분석하는 것이 더 단순해 보이지만, 이번 과제처럼 "필드를 구분해서 저장"하고 "복잡한 집계 쿼리를 빠르게 실행"해야 한다면 RDS가 필요합니다. 그렇다고 RDS만 쓰면 원본 로그를 잃을 위험이 있습니다. 따라서 S3(원본 로그 보관 및 재처리)와 RDS(실시간 분석 및 api 조회)를 함께 두는 것이 실제 프로덕션 환경에서 일반적인 패턴이라고 생각했습니다.

### 3. Lambda vs Fargate

초기에는 Lambda 기반 ETL + 분석까지 고려했지만, analytics.py가 포함하는 작업이 단순 변환이 아니라 SQL 집계 + 시각화 생성까지 포함하면서 Lambda 제약(실행 시간, 패키지 크기)에 걸릴 가능성이 있었습니다.

Fargate는 컨테이너 이미지를 그대로 사용하므로 기존 Docker 파이프라인을 그대로 이식할 수 있고, 실행 시간 제한도 없습니다. 

그래서 분석 로직은 ECS Fargate로 분리했고, Lambda는 S3 이벤트 기반 전처리 역할로 제한했습니다.

### 4. 3가지의 시각화 도구 선택 이유

QuickSight, Streamlit, S3 정적 서빙은 각각 대체 관계라기보다 목적이 다르다고 판단했습니다.

실무 환경에서는 BI 툴 라이선스 비용, 외부 공개 여부, 커스텀 UI 필요도에 따라 QuickSight(관리자용 BI), S3+CloudFront(외부 공유용 정적 이미지), Streamlit(개발자용 인터랙티브) 중 하나를 선택하기에, 하나의 시각화 도구로 통합하기보다는, 사용자의 목적(분석 / 탐색 / 공유)에 따라 결과를 분리 제공하는 구조로 설계했습니다.
